# TheDailyMe — 迭代路线图

> 每个待办事项附「加分标签」说明它在 HR 眼中对应什么能力。
>
> 标签含义：
> - **[工程素养]** — 你不是"能跑就行"，而是有质量意识
> - **[产品思维]** — 你不是在写代码，而是在做产品
> - **[架构能力]** — 你的设计经得起扩展和多人协作
> - **[可靠性]** — 你为项目后果负责，不只管写出来
> - **[技术深度]** — 你理解原理，不只是调包
> - **[影响力]** — 你的项目有外部用户/社区认可
> - **[效率意识]** — 你在意成本、性能、维护负担

---

## Phase 0 — 现在就该补的（2h）

最基础的东西，不补的话面试官可能直接减分。

- [ ] **给核心函数补完整 type hints**
      `generate_daily`、`fetch_all`、`create_sources` 等主要函数的
      参数和返回值都加上类型标注。
      加分：**[工程素养]** 体现你写代码是给别人看的，不是写完就跑。

- [ ] **`print()` → `logging`**
      现在日志混在 stdout 里，没法区分严重级别。改成标准的
      `logging` 模块，INFO 记录正常流程，WARNING 记录源失败，
      ERROR 记录崩溃级异常，日志写文件可追溯。
      加分：**[工程素养]** 维护过线上系统的人都会看重日志规范。

- [ ] **写 10 个 pytest**
      给 `content_engine.py`（`deduplicate`、`score_article`）
      和 `base.py`（`_make_article`）写单元测试。
      加分：**[工程素养]** 有测试的简历和没测试的，面试权重差一个档次。

## Phase 1 — 自动化闭环（2-3d）

让项目从"手工玩具"变成"每天自己跑的服务"。

- [ ] **GitHub Actions 每日定时生成**
      每天 UTC 22:00（北京时间 6:00）自动运行 `daily.py`，
      生成 HTML push 到 `gh-pages` 分支。
      加分：**[可靠性]** 面试官会问"跑了多久、挂了没"——
           你答"89 天零中断，有 3 次源失败但自动降级了"就够了。

- [ ] **GitHub Pages 自动部署**
      生成的 HTML 部署到 `https://heguang-mercy.github.io/TheDailyMe/`，
      别人不用 clone 项目就能看你的日报。
      加分：**[产品思维]** 你做的东西别人能不能用——这是产品经理
           判断你有没有产品直觉的第一问题。

- [ ] **每日日报摘要推送到微信/邮箱**
      用企业微信机器人 / Server酱 / SMTP 之一，每天早上把今日
      头条标题+摘要推到你手机上。
      加分：**[产品思维]** 真正的产品是有人每天在用的。
           你说"我每天早上看这个日报"比说"我有 18 个数据源"有力 10 倍。

- [ ] **Dashboard 页面展示运行状态（在线时长/成功率）**
      Web 客户端加一个状态页：今天生成成功了吗，跑了多久，
      最近 7 天成功率，每个源的健康度。
      加分：**[可靠性]** 你做的东西是不是真能用——健康度页面
           直接告诉面试官你在意服务质量。

---

## Phase 2 — 架构改进（2-3d）

不会在简历上多写一行字，但面试聊到的时候能把你和其他候选人区分开。

- [ ] **采集器自动注册机制（已有装饰器 + 但 `create_sources` 没改）**
      现在 `daily.py` 里的 `create_sources()` 还在用 if-else 链
      判断每个源开不开。改成读取 `@register_source` 装饰器注册的
      信息自动实例化。
      加分：**[架构能力]** "新增一个数据源只需写一个类文件放到
           sources/ 下"——这个设计决策本身就是一个面试高频话题。

- [ ] **采集结果数据库化**
      目前的日报只存 HTML。加 SQLite 存结构化数据：
      `articles(id, title, source, category, url, summary, published_at, collected_at)`。
      加分：**[技术深度]** 提到迁移过程（从无结构到结构化）时
           展示了你理解数据模型演进。

- [ ] **配置从 YAML 迁移到 SQLite DB（多用户前置）**
      为 Phase 3 的多用户做准备，把每个用户的配置存到数据库
      而不是文件，支持动态修改。
      加分：**[架构能力]** "从文件配置到数据库配置"这个跨度说明
           你考虑过规模问题。

- [ ] **Docker 化**
      写 `Dockerfile` + `docker-compose.yml`，一个命令启动
      整个项目。
      加分：**[工程素养]** 2025 年 90% 的简历都说自己会用 Docker，
           但只有 30% 真的写过 Dockerfile 并且理解 multi-stage build。

---

## Phase 3 — 产品化（3-5d）

从"我的代码"变成"一个产品"的分水岭。

- [ ] **多用户系统（简单版）**
      不搞复杂登录。用订阅码/邀请码机制：每个用户一个唯一码，
      填写后生成自己的配置，每天早上收到自己的定制日报。
      加分：**[产品思维]** 你的项目被人用了——哪怕只有 10 个人——
           都比你所有代码加起来更有说服力。

- [ ] **用户反馈通道**
      哪怕是 Google Form / GitHub Issues 模板也行。在日报底部
      加一行"不准确？点这里反馈"。
      加分：**[产品思维]** "用户反馈让我加了 Carbon Brief 源"
           面试官听到这个就知道你不是在闭门造车。

- [ ] **类别权重自动调节**
      根据用户点击/阅读行为自动微调权重。点开科技板块多 →
      科技权重涨。
      加分：**[技术深度]** 做不做成不重要，重要的是你考虑了
           "个性化不是靠用户手动配，而是靠行为学习"。

- [ ] **日报搜索 / 历史检索**
      基于 Phase 2 的数据库，提供关键词搜索历史日报的功能。
      加分：**[产品思维]** 这是一种"场景延伸"——从"每天生产一份"
           到"我三个月前看过一篇关于 XX 的文章"。

- [ ] **RAG：基于历史日报的问答**
      把历史日报文章做 embedding 存向量库，用户问"前天科技板块
      那个关于 GPT-5 的文章说了什么"能直接回答。
      加分：**[技术深度]** 但注意：如果只是"调了 LangChain 接 OpenAI"
           不加分。面试官会问 embedding 模型选了什么、chunk 策略、
           检索为什么用 cosine 而不用 dot product。这些问题答得上
           来才是真加分。

## Phase 4 — 影响力（长期）

这些不是一天能做的，但做到一个就能让简历从"不错"变成"印象深刻"。

- [ ] **写 3 篇博客记录设计决策**
      《为什么我选了 Jinja2 而不是 React？》
      《从 500 行单体到插件架构——TheDailyMe 的重构史》
      《一个个人项目服务了 30 人的教训》
      加分：**[影响力]** 文章发到掘金/知乎/V2EX，面试官很可能
           看到过——你还没面试就已经通过了第一印象。

- [ ] **项目开源后收到外部 PR / Issue**
      有人主动来提功能建议或者 Bug report。
      加分：**[影响力]** 别人愿意为你写代码=你的项目结构和文档
           达到了可协作的标准。

- [ ] **写 ARCHITECTURE.md**
      画出架构图、数据流、核心设计决策和 trade-off。
      加分：**[影响力]** Senior 的基本能力是指点后来者——
           这份文档就是证明。

- [ ] **性能基准报告**
      记录每次采集的耗时、CPU/内存、每个源的响应时间。
      加分：**[效率意识]** 大多数人只管功能，你管质量。面试时
           说"Pitchfork 的 RSS 响应最慢，平均 3.2s，所以我给
           它设了单独的 timeout"比任何抽象的描述都真诚。

- [ ] **接入更多数据源 + 开放数据源贡献指南**
      鼓励别人为你的项目加数据源。提供 `CONTRIBUTING.md`
      说明怎么写一个新采集器。
      加分：**[架构能力] + [影响力]** 设计一个让别人能参与的
           系统，比只自己写难得多。

## 不值得做的事

这几个方向看起来酷，但对简历帮助有限：

| 方向 | 为什么不要 |
|------|-----------|
| 换 React/Vue 重写前端 | Flask + Jinja2 是合理选择。为了"用 React"而重写说明你缺乏技术判断力 |
| 上 Kubernetes | 一个单人项目用 K8s 属于过度设计。面试官会觉得你只会堆技术栈 |
| 上 MongoDB / Redis | 没到那个规模。面试时被问到"为什么选 MongoDB"你答不上来反而减分 |
| 加登录/注册/JWT | 如果只是套个 JWT 教程就加进来，面试官问 OAuth2 流程你接不住 |

## 建议的优先级（性价比排序）

| 优先级 | 事项 | 预估时间 | 加分效果 |
|--------|------|---------|---------|
| P0 | pytest + logging + type hints | 2h | ★★★ 消除减分项 |
| P1 | GitHub Actions 自动跑 + Pages 部署 | 1d | ★★★☆ 有人能看了 |
| P1 | 日报推送（微信/邮箱） | 0.5d | ★★★★ 每天在用 |
| P2 | 采集器自动注册（`create_sources` 重构） | 1d | ★★★☆ 架构加分 |
| P2 | Docker 化 | 0.5d | ★★★ 简历标配 |
| P3 | 多用户 + 反馈通道 | 2d | ★★★★ 产品思维 |
| P4 | RAG / 搜索 | 2d | ★★★☆ 但容易露怯 |
| P5 | 写博客 + 开源 PR | 持续 | ★★★★★ 影响力 |

---

> **一条原则：** 面试官不会因为你功能多而给你加分，
> 但会因为你**为自己的项目负责**而给你加分。
> 100 个数据源不如 1 个真实用户。
> 10 个功能不如 1 篇讲为什么这么做的博客。
